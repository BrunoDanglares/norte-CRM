BLOCO 1 — REMOÇÃO DE CONSOLE.LOG DE DEBUG

1.1 Faça um grep por console.log em todo o projeto (exceto node_modules, dist, .git).

1.2 Para cada ocorrência, classifique:
  - REMOVER: logs de debug óbvios como "aqui", "teste", "funcionou", valores de variáveis sem contexto, logs comentados
  - REMOVER: logs que imprimam tokens, senhas, payloads de webhook completos, dados de usuários
  - MANTER como console.error: erros reais com mensagem descritiva
  - CONVERTER para console.warn: avisos de comportamento inesperado que não são erro
  - MANTER: logs de inicialização de serviço (ex: "Servidor rodando na porta 5000", "Cron de billing iniciado")

1.3 Aplique as classificações. Ao final, informe quantos logs foram removidos, quantos mantidos e quantos convertidos.

BLOCO 2 — TIPAGEM TYPESCRIPT

2.1 Faça um grep por ": any" e "as any" em server/ e client/src/.

2.2 Para cada ocorrência:
  - Se for um cast temporário com comentário justificando → MANTER e adicionar // eslint-disable-next-line se não tiver
  - Se for resultado de JSON.parse sem tipo → substituir por JSON.parse(...) as NomeDeTipo (crie a interface se for simples e óbvia)
  - Se for parâmetro de catch que é tipado como any → correto em TypeScript, MANTER
  - Se for preguiça de tipar um objeto simples → criar interface ou type adequado

2.3 NÃO crie tipos complexos para estruturas que você não conhece completamente. Prefira unknown com asserção explícita a inventar tipos errados.

2.4 Ao final, informe quantas ocorrências de any foram eliminadas e quais foram mantidas com justificativa.

BLOCO 3 — PADRONIZAÇÃO DE IMPORTS

3.1 Verifique se há imports não utilizados em qualquer arquivo .ts ou .tsx. Remova-os.

3.3 Em arquivos TypeScript do backend, verifique se há mistura de import estilo CommonJS (require()) com ESModule (import). Padronize para import ESModule onde o tsconfig permitir.

3.4 No frontend, verifique se há paths absolutos hardcoded (ex: "../../components/ui/button") que poderiam usar o alias "@/components/ui/button". Corrija os casos encontrados.

BLOCO 4 — TRATAMENTO DE ERROS

4.1 Localize todos os blocos catch {} vazios (sem nenhum código dentro). Adicione no mínimo:
  console.error("[NomeDoArquivo] Erro inesperado:", err);

4.2 Localize rotas Express que não têm try/catch. Para cada uma:
  - Se a rota faz operações assíncronas (await) sem try/catch → adicione try/catch com return res.status(500).json({ error: "Erro interno" })
  - Se a rota é síncrona e simples → não precisa de try/catch

4.3 Verifique se há retornos de erro sem status HTTP adequado. O padrão esperado é:
  - 400: dados inválidos/faltando
  - 401: não autenticado
  - 403: sem permissão
  - 404: recurso não encontrado
  - 422: regra de negócio violada (ex: janela 24h fechada)
  - 500: erro interno inesperado
  Corrija casos onde o status está claramente errado (ex: retornar 500 para dado faltando).
